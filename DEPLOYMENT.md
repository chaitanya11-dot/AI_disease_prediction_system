# Deploying to Render

This guide deploys MediSense AI to [Render](https://render.com) using:
- A **Web Service** for the Flask backend (gunicorn)
- A **Static Site** for the React frontend
- A **managed Postgres database** (Render's free tier)

Render's free tier is enough to run this whole app at no cost (with the usual free-tier
caveats: the backend service spins down after inactivity and takes ~30-50s to wake up on the
next request).

---

## Option A — One-click Blueprint deploy (recommended)

This repo includes a `render.yaml` Blueprint that provisions all three pieces together.

1. **Push this project to a GitHub or GitLab repo.**

2. **In the Render dashboard:** New → Blueprint → connect the repo. Render will detect
   `render.yaml` and show you a preview of what it's about to create:
   - `medisense-db` (Postgres, free)
   - `medisense-backend` (web service)
   - `medisense-frontend` (static site)

3. Render will prompt you to fill in the variables marked `sync: false` before the first
   deploy. At minimum, set on **medisense-backend**:
   - `ADMIN_EMAIL` / `ADMIN_PASSWORD` — your own admin login (don't leave these blank in
     a real deployment)
   - `CORS_ORIGINS` and `FRONTEND_URL` — you can fill these in after the frontend's first
     deploy once you know its URL (see step 5)

   `SECRET_KEY` and `JWT_SECRET_KEY` are auto-generated by Render (`generateValue: true`) —
   you don't need to set these yourself.

4. Click **Apply**. Render will:
   - Provision the Postgres database
   - Build the backend (`pip install -r requirements.txt && python ml/train_model.py`)
   - Run `start.sh` (seeds the DB, then starts gunicorn)
   - Build the frontend (`npm install && npm run build`) and serve `dist/`

5. **Connect the two services to each other** (one-time, after first deploy):
   - Copy the backend's URL, e.g. `https://medisense-backend.onrender.com`
   - Go to **medisense-frontend → Environment**, set:
     `VITE_API_URL=https://medisense-backend.onrender.com/api`
   - Trigger a manual redeploy of the frontend (env var changes require a rebuild for a
     static site, since Vite bakes the value in at build time)
   - Copy the frontend's URL, e.g. `https://medisense-frontend.onrender.com`
   - Go to **medisense-backend → Environment**, set:
     `CORS_ORIGINS=https://medisense-frontend.onrender.com`
     `FRONTEND_URL=https://medisense-frontend.onrender.com`
   - Save — the backend will redeploy automatically

6. Visit your frontend URL. Log in with the `ADMIN_EMAIL` / `ADMIN_PASSWORD` you set in step 3.

## Option B — Manual setup (without the Blueprint)

If you'd rather click through the dashboard manually instead of using `render.yaml`:

### 1. Create the database
New → PostgreSQL → name it (e.g. `medisense-db`) → free plan → Create.
Copy the **Internal Connection String** once it's ready.

### 2. Create the backend web service
New → Web Service → connect your repo.
- **Root directory:** `backend`
- **Runtime:** Python 3
- **Build command:** `pip install -r requirements.txt && python ml/train_model.py`
- **Start command:** `bash start.sh`
- **Environment variables:**
  | Key | Value |
  |---|---|
  | `DATABASE_URL` | (paste the Postgres connection string from step 1) |
  | `SECRET_KEY` | (generate: `python -c "import secrets; print(secrets.token_hex(32))"`) |
  | `JWT_SECRET_KEY` | (generate another one the same way) |
  | `RENDER` | `true` |
  | `ADMIN_EMAIL` | your chosen admin email |
  | `ADMIN_PASSWORD` | your chosen admin password |
  | `CORS_ORIGINS` | (fill in after step 3, see below) |
  | `FRONTEND_URL` | (same as `CORS_ORIGINS`) |

### 3. Create the frontend static site
New → Static Site → connect your repo.
- **Root directory:** `frontend`
- **Build command:** `npm install && npm run build`
- **Publish directory:** `dist`
- **Redirect/rewrite rule:** add `/*` → `/index.html` (Rewrite), so React Router's
  client-side routes don't 404 on refresh.
- **Environment variable:** `VITE_API_URL` = `https://<your-backend-service>.onrender.com/api`

### 4. Wire them together
Once both are deployed, go back to the backend service's environment variables and set
`CORS_ORIGINS` and `FRONTEND_URL` to the frontend's actual `.onrender.com` URL, then save
(triggers a redeploy).

## Notes specific to this app

- **The ML model ships pre-trained**, but the Blueprint's build command retrains it anyway
  (`python ml/train_model.py`) so the model files are always freshly generated for the exact
  scikit-learn version installed on the build machine — pickle files aren't guaranteed
  compatible across differing library versions, so this avoids a subtle class of bugs.
- **`start.sh` runs `seed_db.py` on every boot**, not just the first deploy. This is
  intentional and safe — `seed_db.py` skips records that already exist — but it does mean
  changing `ADMIN_EMAIL`/`ADMIN_PASSWORD` after the first deploy won't retroactively change
  an already-created admin account. To rotate the admin password later, log in as that admin
  and use the in-app "Change password" flow on the Profile page instead.
- **Email stays in console-mode** (printed to Render's log viewer) unless you set `MAIL_SERVER`,
  `MAIL_USERNAME`, and `MAIL_PASSWORD`. A free Gmail account with an
  [App Password](https://support.google.com/accounts/answer/185833) works fine for low volume.
- **Free-tier cold starts:** Render's free web services sleep after ~15 minutes of inactivity.
  The first request after a sleep can take 30-50 seconds while it wakes up — this is normal,
  not a bug in this app.
- **Render's free Postgres expires after 90 days** unless upgraded to a paid plan — keep this
  in mind for anything beyond a demo/portfolio deployment.
